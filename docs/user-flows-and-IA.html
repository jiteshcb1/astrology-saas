Edge cases to design for
  • Slot taken by someone else during checkout → re-prompt with fresh availability
  • Payment succeeds but webhook is delayed/missed → reconciliation job confirms later; show "confirming…" state
  • Payment fails or is abandoned → hold slot briefly, then release; allow retry
  • No team member free at that slot (round-robin) → slot shouldn't have been offered; transactional re-check
  • Seeker closes tab after paying → confirmation email still fires (idempotent)
  • Duplicate submit / double-click → idempotency prevents two bookings

  • Fake / wrong proof uploaded → consultant rejects; booking stays unconfirmed; seeker notified
  • Consultant never confirms → auto-expire after N hours, release slot (Phase 1.5)
  • Seeker paid but forgot to upload proof → reminder; manual reconciliation

Team Member Flows

Two roles, strictly scoped. Role isolation is a security requirement, not just a UI choice.

Consulting members must NOT see account-wide payments. Accounts members must NOT see call content/notes. Enforced server-side, not just hidden in UI.
Flow 1 — Consulting member runs assigned calls
Accepts invite + first login→Sets own availability→Receives auto-assigned bookings
↓
Opens assigned booking→Reads seeker questions→Joins Meet→Writes notes / uploads chart→Recorded ✓
Edge cases
  • Tries to open a booking not assigned to them → blocked (server-side)
  • Tries to view payments → not in their nav, and API-denied
  • Goes unavailable → round-robin stops assigning to them
Flow 2 — Accounts member manages money
Accepts invite + first login→Sees payments list→◆ Pending UPI-QR proof?
Yes
Verifies proof → confirms→Booking confirmed + receipt→Done ✓
↓
Reviews receipts & GST→Reconciles
Edge cases
  • Tries to open call notes → not in their nav, API-denied
  • Confirms a payment then disputes → audit trail of who verified

Super Admin (Operator) Flows

Your control plane — low volume, high stakes. Mostly management + read-only oversight.

Flow 1 — Onboard a consultant manually
Super Admin console→Create organization + owner→Assign a plan + seats→Send invite email→Consultant activated ✓
Edge cases
  • Email already exists → conflict handling
  • Consultant never activates → pending org state
Flow 2 — Manage subscription & suspension
Consultant detail→◆ Payment status?
Lapsed
Dunning → grace period→Suspend org→Public page offline; calls retained
Recovered
Reactivate→Page live again ✓
Edge cases
  • Suspend org with upcoming paid bookings → keep confirmed calls honored
  • Downgrade plan below current seat count → require seat removal first
Flow 3 — Feature flags & oversight
Console→Toggle feature (global/plan/org)→Takes effect without deploy
↓
Open oversight (read-only)→View any org's bookings/receipts→For support/compliance ✓
Edge cases
  • Flag precedence conflict → org > plan > global resolves it
  • Oversight must stay read-only → no accidental edits to tenant data
